## 一、系统概述

### 1.1 系统简介

游戏化激励系统是汽车维修店内部管理系统的核心模块之一，通过段位管理、激励规则配置、红包奖励、勋章系统、积分商城等功能，提升员工工作积极性和门店运营效率。

### 1.2 核心功能

- **段位管理**: 门店段位、区域段位的评级与排名
- **激励规则**: 灵活配置的激励规则引擎，支持多种主题类型
- **红包系统**: 红包账户、提现审核、发放记录
- **勋章系统**: 勋章发放、回收、积分换算
- **标签配置**: 商品、门店、员工标签的灵活配置
- **积分商城**: 积分兑换商品的管理与运营

### 1.3 技术栈要求

- **后端框架**: Spring Boot + MyBatis
- **数据库**: MySQL 5.7+
- **文件存储**: OSS（阿里云/腾讯云）
- **定时任务**: Spring Task / XXL-Job
- **缓存**: Redis（排名、统计数据）
- **消息队列**: RabbitMQ / Kafka（异步处理）

### 模块3：门店C端激励规则配置 ⭐⭐⭐⭐⭐

> **核心复杂模块**: 开发工作量最大，业务逻辑最复杂

#### 3.1 新增激励规则

**基本信息**:
- 规则名称
- 规则编码
- 激励方式：拇指积分、红包、勋章等
- 业务分类：点检、行为力填报、实名认证、接车单、技师接车单、配件等
- 发放来源：仅拇指积分可修改

**主题类型**（去掉连续签到天数）:
1. **普通规则**: 标准激励规则
2. **梯度累计**: 达到不同梯度累计发放
   - 示例：售卖5条轮胎发1个红包，10条、15条继续发放
3. **破冰**: 第一次达到每个梯度只发放1次
4. **月首单**: 每月第一单奖励
5. **历史首单**: 一段时间内第一单奖励
6. **每周之星排名**: 排名前N名奖励
7. **每月之星排名**: 排名前N名奖励

**规则配置**:
- **排名分组**: 按大区、按区域、按员工等级
  - 示例：统计傲世途排名，有30个大区组，在我的大区组中排名
- **员工标签**: 多选，关联报名比赛活动的员工
- **门店标签**: 选中特定类型门店（非筛选，是选中）
- **是否互斥**: 
  - 互斥：优先首单奖励（示例：4条轮胎，2条首单+2条普通）
  - 不互斥：叠加发放（示例：2条首单+4条普通）
- **账号数量限制**: 门店多少人首单有奖励
- **配件数量限制**: 前N个配件为首单红包，后续为普通红包

**延保相关**（仅接车单/技师接车单）:
- **是否延保**: 是否与延保关联（默认否，必填）
  - 否：接车单结案直接发放
  - 是：完成接车单+延保两个条件同时满足
- **是否发放延保注册人**: （默认否，必填）
  - 否：发给接车人
  - 是：发给延保注册人（可能不是同一人）

**勋章图标**:
- 仅上传1个
- 未上传显示上传按钮
- 已上传显示图片，点击可放大

**统计分组**:
- 目前后台写死数据
- 后续支持可配置（商品标签中的）
- 需与客户讨论方案

**适用范围**:
- 服务门店（系统预设）
- 全部门店（系统预设）
- 门店标签（配置的标签）
- 单独选择的门店
- 门店标签和单独门店同时生效

#### 3.2 编辑激励规则

**状态控制**:
- 外层状态为"停用"，才能增加、删除操作
- 外层状态不影响修改启用状态
- 停用状态编辑时有保存按钮
- 启用状态编辑页面无保存按钮

**编辑权限**:
- 所有字段可编辑（状态为停用时）
- 延保相关配置可修改

#### 3.3 设置随机红包

**配置内容**:
- 红包金额区间（最小值-最大值）
- 占比设置（百分比）
- 当前平均成本计算

**成本计算公式**:
```
当前平均成本 = Σ((区间最小值 + 区间最大值) / 2) × 占比
示例: 3×60% + 7.5×40% = 4.8元
```

**页面提示**:
- 需添加备注说明
- 显示实时平均成本

#### 3.4 删除激励规则

**删除条件**:
- 仅停用状态可删除
- 已有发放记录的规则提示确认

#### 3.5 禁用/启用规则

**状态切换**:
- 启用 ↔ 禁用
- 禁用后不再触发激励
- 已发放记录不受影响

#### 3.6 审核激励规则

**审核逻辑**:
- 停用状态才能审核
- 支持批量审核
- 双击或点击审核按钮打开审核弹窗

**审核内容**:
- 规则配置检查
- 统计分组验证
- 适用范围确认

**数据表**:
- `acct_subsidy_rule` (红包规则主表)
- `acct_subsidy_rule_grade_detail` (红包规则明细表)
- `acct_subsidy_rule_area` (红包规则使用范围表)
- `acct_subsidy_rgd_price_range` (红包规则随机金额设置表)
- `acct_subsidy_tag` (激励标签主表)
- `acct_subsidy_tag_area` (门店类型标签范围明细表)

- 点击图片可放大显示

**范围类型**:
- 全国
- 省份
- 城市
- 门店
- 总部

**排序规则**:

- 分类一样时无所谓排序

**商品状态**:
- **未上架**: 保存后默认状态
- **上架**: 可兑换
- **下架**: 不可兑换

- ✅ 下架商品可编辑
- ✅ 未上架商品可编辑
- ❌ 上架商品不能编辑（需先下架）

#### 8.3 上架商品

**上架条件**:
- 商品信息完整
- 库存 > 0
- 状态为未上架或下架

#### 8.4 下架商品

**下架操作**:
- 上架状态可下架
- 下架后不影响已兑换订单

#### 8.5 同步外部商品

**同步功能**:

## 七、风险与注意事项

### 7.1 关键风险点

#### 1. 模块3（激励规则配置）复杂度极高 ⚠️

**风险描述**:
- 工作量占比23%（11人天）
- 业务逻辑最复杂
- 7种主题类型的不同处理逻辑
- 互斥规则、梯度累计、排名分组等复杂算法

**应对措施**:
- 安排资深开发负责
- 提前与客户确认业务规则
- 分主题类型逐步开发测试
- 充分的单元测试覆盖

#### 2. 业务规则需反复确认 ⚠️

**需要与客户确认的关键点**:
- 综合分计算公式
- 统计分组配置方式
- 定时统计时间点
- 红包互斥逻辑细节
- 排名分组算法
- 梯度累计触发条件

**应对措施**:
- 开发前召开业务需求评审会
- 每个关键算法都要书面确认
- 保留配置化空间，避免硬编码

#### 3. 数据表关联复杂 ⚠️

**复杂关联**:
- 激励规则关联8张表
- 多表联查可能影响性能
- 数据一致性要求高

**应对措施**:
- 合理设计索引
- 使用Redis缓存热点数据
- 排名数据定时计算，不实时查询
- 考虑读写分离

#### 4. 并发与性能问题 ⚠️

**潜在问题**:
- 红包发放并发冲突
- 积分变动数据一致性
- 排名查询性能
- 大数据量统计慢

**应对措施**:
- 红包账户使用分布式锁
- 积分变动使用乐观锁
- 排名使用Redis Sorted Set
- 大数据统计使用定时任务 + 结果缓存

#### 5. 定时任务开发 ⚠️

**需要的定时任务**:
- 段位统计任务（月度/周度）
- 积分结算任务
- 排名更新任务
- 数据归档任务

**应对措施**:
- 使用XXL-Job统一管理
- 任务失败告警
- 支持手动补跑
- 任务执行日志记录
